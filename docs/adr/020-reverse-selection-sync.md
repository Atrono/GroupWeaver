# ADR-020: Reverse sidebar→graph selection sync — projecting a selection onto the canvas

**Status:** Accepted · **Date:** 2026-06-22
**Refines:** ADR-018 (which shipped the graph→VM half and **split this reverse sync out** — its Consequences note: "The **reverse sidebar→graph selection sync** is deferred to its own issue (C# VM change)") · ADR-004 D5 (renderer seam) · ADR-007 D1 (selection is never busy-gated) · ADR-010 §5 (jump-to-node) · **Decides:** issue #96 (the reverse selection sync) · **Phase:** 4 (v0.2 polish pass)

## Context

ADR-018 D3 shipped the graph→VM direction: a node **tap** sends `nodeClick` (the unconditional first statement), then `applySelection` paints `node:selected` + the 1-hop neighborhood dim. It **split out** the reverse — a selection originating in the native chrome (a sidebar row jump, or any other `SelectedDn` set) never reached the canvas, so the `:selected` border and neighborhood dim only appeared on a graph tap. ADR-018 deferred it explicitly because it needs a VM→JS `select` command and touches the pinned jump/selection-sync C# tests: in particular `JumpCommand_SetsSelectedDn_AndFocusesTheAnchor` asserts `FocusCalls` increments by **exactly 1** per jump, so the new selection projection must not ride the focus channel.

## Decision

1. **`SelectAsync(string dn, …)` as a default no-op interface method** on `IGraphRenderer`, mirroring ADR-019's `SetBusyAsync` (and `ShowDiffGraphAsync`/`ExportPngAsync`) — so `FakeGraphRenderer` and every implementation compile without an override, and only the real `CytoscapeGraphRenderer` overrides it. Drives `node:selected` + neighborhood dim on the node named `dn`; an empty `dn` clears the selection.

2. **Dispatched fire-and-forget, off every shared channel — its own renderer `select` command, NEVER `FocusCalls`.** This is the load-bearing reason for a *separate* channel: `JumpAsync` both sets `SelectedDn = dn` (→ `select`, via `OnSelectedDnChanged`) **and** calls `FocusAsync([dn])` (→ `focus`), and the pinned `JumpCommand_SetsSelectedDn_AndFocusesTheAnchor` asserts `FocusCalls` increments by **exactly 1** per jump. Routing the selection projection through `FocusAsync` would double-count it and break the pin. The real override is structurally identical to `SetBusyAsync`: **no** `EnterSingleFlight`/`_commandInFlight` (the caller is the selection-change hook, never busy-gated, ADR-007 D1 — it must stay responsive during an in-flight pipeline), **no** confirmation TCS (never the 60 s `BridgeTimeout`, never the `focused` channel), a **non-blocking** ready check (`_webView is null || !_ready.Task.IsCompletedSuccessfully` → return), and **never-throw** (the discarded `_ =` task has no handler; a degraded bridge surfaces as `RendererError` and returns normally). The `select` command is serialized through `GraphJson` (a `SelectDto` record beside `FocusDto`/`BusyDto`) so a comma/quote-containing DN is escaped exactly as a focus DN is.

3. **Wired from `OnSelectedDnChanged`, NOT `JumpAsync`.** Hooking the property-change partial — not just the jump command — means **every** selection-change source (a sidebar-row jump, a future detail-panel select, and a graph tap that round-trips `SelectedDn` back) syncs the canvas uniformly. It is **not busy-gated** (ADR-007 D1: selection re-projection stays responsive mid-pipeline, the same contract `RecomputeDetailPanel`/`HighlightActiveRows` already follow on this hook). `partial void` cannot be async, so the never-throw task is discarded with `_ =`.

4. **graph.js `case 'select'` reuses `applySelection`/`clearSelection`** (the same functions a tap uses), so a command-driven select is **byte-identical** to a tap-driven one — same `node:selected` + neighborhood dim. **INSTANT** (`addClass`/`removeClass` only, never `cy.animate`), so the #88 isolated motion counters (`__gwAnimateCalls`/`__gwEnterAnims`) stay 0. `cy === null` → silent break (no `jsError`, keeping the `verify.mjs` zero-jsError audit green for a benign pre-commit race, as `busy` does). `getElementById` only (comma-DN safe, ADR-004 D5).

5. **Empty/unknown id → `clearSelection`; null value → clears; redundant select after a tap is idempotent.** A `null` `SelectedDn` dispatches an empty DN → `clearSelection`, so dropping the selection in the chrome visibly clears the canvas. An unknown DN (not on the canvas) resolves to an empty collection → `clearSelection`, surfaced as a clear rather than a stale highlight. A tap already paints via `applySelection`, then `SelectedDn` round-trips and re-issues a redundant `select` — harmless and **intentionally unguarded** (idempotent; guarding it would re-introduce tap/select coupling ADR-018 D3 deliberately avoided).

## Consequences

- A sidebar jump (and any non-tap selection) now drives the canvas `:selected` border + neighborhood dim, closing the ADR-018 gap; the graph acknowledges selections from both directions.
- The **`SelectCalls`-not-`FocusCalls` split** keeps the `JumpCommand_SetsSelectedDn_AndFocusesTheAnchor` pin green: the jump still focuses exactly once, and now also selects exactly once on a separate channel.
- `verify.mjs` (owned by the test-engineer) gains `select`-command asserts — parity with a tap (byte-identical selection + dim), clear-on-empty, and motion-counters-untouched (`__gwAnimateCalls`/`__gwEnterAnims` must not move on a `select`). `WebBundleTests` gains a `'select'` wire tripwire (the new command case must exist in the bundle).
- `FakeGraphRenderer` needs a `SelectAsync` override recording its own `SelectCalls` channel (never `FocusCalls`) for the `WorkspaceViewModel` selection/jump tests — a `tests/` change owned by the test-engineer.
