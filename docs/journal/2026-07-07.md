# 2026-07-07 - Session 44 (minimap click-pan fix, #280 closed)

**Phase:** Phase 4 (feedback-driven). Follow-on from session 43's fit-audit batch: fixed
the one bug found along the way but filed separately - the minimap's mouse-based
click/drag-to-pan never actually targeted the clicked point.

## Done

- **PR #284 (squash `cfe4119`), closes #280.** `minimapPanTo` called
  `cy.center({position:{x,y}})`, but cytoscape only treats a selector function or an
  existing collection as valid `center()` input - a plain `{position}` object silently
  fell through to centering on the whole graph's bounding box (`mutableElements()`),
  ignoring the clicked point entirely. The feature had likely never worked correctly
  since it shipped (WP3d, #146) - idempotent after the first click, since every
  click re-centered on the same whole-graph bbox. Replaced with a direct `cy.pan()`
  computation (`pan.x = width/2 - mx*zoom`, `pan.y = height/2 - my*zoom`), mirroring
  the sibling `minimapPanByKey`'s (#279) rendered-pixel-space convention - `cy.pan()`
  has no selector/collection resolution path, so it can't silently no-op the way
  `center()` did. New `verify.mjs` assertion dispatches two well-separated synthetic
  minimap clicks and confirms each lands the viewport on its OWN target, not the graph
  center or the other click's target - verified to genuinely FAIL against the reverted
  buggy code before confirming it passes with the fix. Gate green (858 Core + 1177
  App), reviewer APPROVE (independently verified the pan-centering math and the
  browser's `MouseEvent.clientX` integer-truncation behavior), CI green.
