# Design — v0.2 Polish Pass (audit-first)

**Date:** 2026-06-21 · **Phase:** 4 (feedback-driven) · **Status:** approved, executing
**Origin:** brainstorming session — *"optimize UI/UX first, or build v0.4 (Entra ID)?"*

## Goal

Make the **already-shipped v0.2 product feel genuinely great** — lift **visual identity**,
**interaction feel / motion**, and **edge-case & state UX** from *correct* (the screenshot
gates already prove this) to *crafted*. Not features (no v0.4), not adoption mechanics.

The leverage stop-rule bounds gold-plating on low-traffic surfaces; on the **core surfaces
(graph + workspace) the bar is "great," not "good enough."** The `frontend-design` skill —
"distinctive, intentional visual design… choices that don't read as templated defaults" —
is the named lens for the visual-identity work, applied in the audit and driving every
identity slice.

## Why polish, not v0.4

v0.4 (Entra ID / M365) would break the core **E2** decision (on-prem LDAP + integrated
Windows auth, *no credential handling*) and is the scope-creep risk PLANNING.md §8 names.
No user is pulling for M365 (one open issue, #54, self-filed + deferred). Polishing the
shipped product is the higher-leverage, lower-risk move while Phase 4 waits on real feedback.

## Hard constraint — the visual language is *pinned*

Exact hex values (kind palette, severity `#D13438/#F7A30B/#4FA3E3`, diff cues) carry
**C#↔JS parity tests** and render in four places at once — graph canvas, native chrome,
legend overlay, detail badge. Any identity change touching color/type is a coordinated,
test-re-pinning change needing a single source of truth — deliberate, never a free edit.

## Standards & design anchors (currently implicit — make them explicit)

The repo has good instincts encoded but **no declared standards**. Declaring them is the
backbone that makes "great" objective, not subjective. Phase 0 declares, Phase 1 builds,
Phase 2 pins.

- **Brand (to articulate):** emergent today — Fluent-aligned dark, semantic palette as
  pinned C#+JS constants (`AdObjectKindConverters.cs` / `SeverityConverters.cs` /
  `GapKindConverters.cs` / `web/graph.js`), **no tokens file, no brand doc**. Define a
  one-paragraph identity (*calm, governance-grade, trustworthy, dense-but-legible*) + a
  **single-source token set**. ADR-gated (parity contracts).
- **UI standard = WCAG 2.2 AA:** 1.4.3 text contrast (4.5:1), 1.4.11 non-text/graphical
  contrast (3:1 — node/edge/halo distinguishability), 1.4.1 use of color (already met —
  severity & diff carry a redundant non-color channel; keep it), 2.4.7 focus-visible. The
  Phase-0 contrast script makes it *measured*. **Honest limit:** full keyboard operability
  (2.1.1) of the Cytoscape canvas is hard — target documented partial conformance.
- **UX lens = Nielsen's 10 heuristics** + project-native principles: **honesty**
  (null-vs-empty tri-state, "unexpanded areas are unchecked"), **airspace** (nothing layers
  over the graph), **error-prevention-by-design** (read-only, gated root pick),
  **recognition over recall** (same kind = same color everywhere). Named acceptance
  criteria, not vibes.

## Carried-in non-negotiables

- **Read-only product** — polish is presentation-only; zero AD-write paths.
- **Public media = demo mode only** — any new GIF/screenshot is `--demo`, never lab AD.
- **Don't regress** the screenshot-gated correctness items; the elevated bar is additive.
- Parity tests stay green, or are deliberately re-pinned in a reviewed PR.

## Intersections to account for (constraints, not new features)

- **i18n / localization:** state-UX polish is copy work. Shipped strings are English; the
  lab DC is German-localized with long German names/DNs — layout pressure
  (wrapping/truncation). Don't build i18n; don't harden English past a future string-resource
  seam; prove layout holds under long/German text.
- **Theme scope:** make dark-only an explicit policy; the token set should *enable* a future
  light theme, never foreclose it.
- **Motion vs. perf floor:** measure animations in **software rendering** (the RDP/server/VM
  floor) within the 5k-node budget. A transition that tanks the floor is a regression.
- **Security of user-data surfaces:** detail-panel attributes, settings notes, naming
  preview, plan-editor names render untrusted strings under **plain-text / no-control-char**
  guarantees (#45, U+2018 audit). Polish must not reintroduce interpretation; such changes
  trigger `/security-review`.
- **Cross-mode consistency:** the three dimensions cover **Plan Mode + Gap Mode** too
  (`plan-editor`, `gap-view` fixtures), not just the Ist workspace.
- **Preserve accessibility redundancy:** any recolor must KEEP the non-color-only channels
  (severity letter+padding, diff brightness+line-style) — stripping them breaks WCAG 1.4.1.

## Phase 0 — Audit (→ the backlog)

0. **Motion-evidencing tooling first** (the one real capability gap — `[I]` items can't be
   judged from static frames; no interactive-input desktop + one-frame PrintWindow lag).
   Extend `tools/capture-window.ps1` into a **timed frame-burst / short video** of the
   windowed `--demo` run while posting `WM_*` input (drag, wheel-zoom, dbl-tap expand,
   selection). Add a **WCAG contrast-ratio script** over rendered PNGs. State the
   **rendering mode** (GPU vs software) on every motion observation. Reviewed slice.
1. **Render every shipped surface** with existing fixtures: native chrome
   (`ShellScreenshotTests` → 32 PNGs), graph layer (`tools/test-graph-bundle.ps1` /
   `verify.mjs` → 5 PNGs), live feel (windowed `--demo` via `capture-window.ps1`).
2. **Judge against the elevated bar** across the three dimensions (identity via
   `frontend-design`; motion via the `[I]` items on the live run; edge-state UX across every
   empty/error/loading/honesty state), measured against WCAG 2.2 AA + Nielsen heuristics.
3. **Locate the palette single-source** (where C#↔JS parity is asserted) — decides whether
   identity gains need recoloring at all.
4. **Decide the visual-identity ADR gate:** palette/token changes → ADR-gated design-tokens
   slice first; type/spacing/motion-only → no ADR, proceed.
5. **Emit the ranked backlog** — each item `{surface, dimension, current-state (screenshot
   ref), proposed refinement, effort S/M/L, leverage}` — as GitHub issues + a summary doc.

## Phase 1 — Execute prioritized slices (incremental, normal gate)

Small vertical slices, subagent-driven (`implementer` / `ui-verifier` / `test-engineer` /
`reviewer`). Each through the full DoD: `pwsh tools/build.ps1` green → `ui-verifier` PASS →
`reviewer` APPROVE → push → `ci-sentinel` green → close the issue. Re-screenshot each slice;
raise `docs/ui-checklist.md` when the bar moves up. Motion items judged from the windowed
`--demo` capture, recorded in the PR body. **Re-evaluate leverage every 2–3 slices; stop
when it drops.** No fixed Phase 2/3 of polish — the backlog defines the slices; this is a
loop, not a sequence.

## Phase 2 — Consolidate & ship (closing)

Pin the elevated bar into `docs/ui-checklist.md` (+ parity pins / baselines); refresh public
demo media only if visuals changed; reconcile README / CHANGELOG `[0.2.x]` / PLANNING.md;
cut `0.2.x` **on the user's tag call** (bump `Directory.Build.props` + SmokeTests pin →
`/security-review` if an input surface changed → tag → `release.yml` attests/publishes →
verify the public zip). No release cut unprompted.
