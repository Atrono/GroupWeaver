# GroupWeaver - Full-App UX Fit-Audit (2026-07-01)

> Generated by the `fit-audit` workflow (run `wf_73bbd202-f89`): 9 surfaces, one critic per surface -> two diverse-lens jurors (consumer persona + WCAG/ADR correctness) -> completeness critic -> synthesizer.
> **Read-only / advisory** - every proposed fix is presentation-only; no write path. This doc is the persisted output; levers are candidates for GitHub issues.
>
> **Surfaces:** connect, rootpicker, workspace, graph, audit, plan, gap, settings, crosscut &nbsp;|&nbsp; **Findings:** 59 &nbsp;|&nbsp; **Levers:** 5

## Verdict

GroupWeaver's core product surfaces (graph layer, audit honesty model, rule engine, detail panel) are mature and the v0.2 polish bar is largely met — the redesign initiative's big levers (band severity-gate, focus-visible ring, adaptive rail, theme retone) have landed as Accepted ADRs. The 59 findings are overwhelmingly last-mile fit, not architecture: they cluster into five tight, read-only-safe themes. The genuinely load-bearing gap is accessibility of glyph-only native controls (theme toggle, help '?', rail chevron have a ToolTip but no AutomationProperties.Name, so a screen reader announces the raw symbol — a real WCAG 4.1.2 fail, trivially fixed). Second is broken action-hierarchy in three secondary surfaces (Root Picker, Plan, Settings File) where the app's own checklist bars two-identical-grey-buttons yet they ship. Third, three graph overlays (command palette, minimap, issues-only filter) shipped without a checklist row or an Accepted ADR and carry ARIA-completeness gaps — an unjudged surface, which is a process as much as a UX miss. The remaining two levers are contrast/brand hygiene (off-brand Fluent-blue selection bands vs the ADR-026 purple, sub-3:1 kind badges missing the lift-ring the graph layer already got) and meaning leaking to caption-opacity/translucent fills in the Audit ring and tiles. Nothing requires a new write path; every fix is presentation-only. One finding (native-chrome reduced motion) is already settled Deferred by Accepted ADR-034 and must not be re-litigated. Overall fit: good foundation, a focused accessibility + hierarchy pass closes the visible gap between the signature graph and the surrounding chrome.

## Top 5 levers (ranked)

### Lever 1 - Glyph-only native controls expose no accessible name (tooltip is not a name)

- **Problem:** The theme toggle, the '?' keyboard-help button, the workspace rail-collapse chevron, and the focus/settings glyph pairs carry only ToolTip.Tip and a raw glyph as content. In Avalonia a tooltip is NOT the accessible name, so a screen reader announces the bare symbol (or a missing-glyph box) instead of the action. This is the single load-bearing WCAG 4.1.2 fail across the native chrome, and it recurs on every step.
- **Code:** src/App/Views/MainWindow.axaml:134-138 (theme toggle), :149-150 (keyboard-help '?'), :139-142 (focus mode); src/App/Views/WorkspaceView.axaml:80-87 (rail chevron)
- **Impact:** A keyboard/AT auditor — the product's own persona — cannot identify these controls; the theme toggle and help entry are announced as punctuation. Native chrome fails WCAG 2.2 SC 4.1.2 Name, Role, Value and SC 2.4.4.
- **Fix (read-only):** Add AutomationProperties.Name (mirroring the existing tooltip text) to each glyph-only Button; keep the tooltip. Presentation-only, no behaviour or write path. Pair with a read-only F1/'?' keydown in MainWindow.OnKeyDown that invokes the existing OpenKeyboardHelpCommand so the help window is keyboard-reachable (crosscut-3).
- **Subsumes:** crosscut-1, crosscut-2, crosscut-6, crosscut-3, miss-focus-mode-btn-name, connect-1

### Lever 2 - No action hierarchy on secondary surfaces — the two-identical-grey-buttons ambiguity the checklist bars

- **Problem:** Root Picker (Back/Load), Plan mode (every header + per-form button), and the Settings File tab (Import/Export/Reset) render as identical default-Fluent grey buttons with no accent-primary / ghost-secondary distinction — directly violating the checklist's own Action-hierarchy clause (line 118) that Connect and Settings Save/Apply already honour.
- **Code:** src/App/Views/RootPickerView.axaml:56-57; src/App/Views/PlanView.axaml:57-71,95,142,188-190,206,236; src/App/Views/SettingsWindow.axaml File tab (~line 810-821)
- **Impact:** The primary action (Load a scope / Add / Apply JSONC) is indistinguishable from destructive-or-secondary siblings, so the eye has no anchor — reads as a template, not the intentional product the rest of the app achieves.
- **Fix (read-only):** Apply the existing Classes="accent" to the one primary action per surface and Classes="ghost" (or accent-outline) to the secondaries, reusing the already-shipped button class styles. Pure XAML class assignment, no new tokens, no write path.
- **Subsumes:** rootpicker-1, plan-1, settings-2, plan-5

### Lever 3 - Three graph overlays are unjudged (no checklist row, no Accepted ADR) and carry ARIA-completeness gaps

- **Problem:** The command palette (WP3c/#144), minimap (WP3d/#146), and issues-only filter shipped in src/App/web without a ui-checklist §A clause and without an Accepted ADR (ADR-023 covers only the find/controls cluster). The palette combobox is also an incomplete ARIA pattern (no aria-activedescendant; option <li> rows have no id), find/'no issues' status changes are not announced (no aria-live/role=status), and light-theme 'No match' text measures 3.44:1 (< 4.5:1).
- **Code:** src/App/web/index.html:174-374 (#palette-results, #minimap, role=combobox), src/App/web/graph.js:~1344 (aria-selected set but no id/activedescendant); docs/ui-checklist.md §A (no palette/minimap/issues-filter rows)
- **Impact:** An entire interactive surface is outside the DoD's declared bar (CLAUDE.md step 2 judges vs the matching checklist section), and AT users get an incomplete combobox with silent status changes; the light-theme no-match string fails SC 1.4.3.
- **Fix (read-only):** Read-only, in-scope: (a) add ui-checklist §A rows for the palette, minimap, and issues-only filter and record their design in a Proposed ADR (adr-author skill); (b) complete the combobox — give each option <li> an id and set aria-activedescendant on the input to the active row; (c) wrap the no-match/'no issues' text node in an aria-live=polite/role=status region; (d) bump the light-theme no-match colour token to clear 4.5:1 (verify with tools/check-contrast.ps1). All presentation/markup, no write path.
- **Subsumes:** graph-1, graph-2, graph-4, graph-5, graph-3, graph-7, graph-6

### Lever 4 - Off-brand selection accent and sub-3:1 kind badges in the list/detail surfaces

- **Problem:** Selected-row highlight in the Root Picker and the Gap changes list uses default Fluent accent blue instead of the ADR-026 brand purple (AccentFillHex #6B5BDF / AccentHex #8B7BFF), and the DL/UG kind-badge fills in the picker and detail-panel header fail WCAG 1.4.11 3:1 against the page — the picker/detail surfaces never got the 2px #8A93A3 contrast-lift ring the graph layer already carries for exactly these fills, and the detail badge hardcodes white ink with no per-hue contrast pin.
- **Code:** src/App/Views/RootPickerView.axaml (selected-row highlight + kind badges), src/App/Views/GapView.axaml (active-row band), src/App/Views/DetailPanelView.axaml (header kind badge ink); BrandTokens.cs:231-281 (AccentFillHex/AccentHex)
- **Impact:** The redesign's brand accent is inconsistent (blue islands in a purple app), and low-contrast badges are hard to distinguish for low-vision users — the same defect the graph layer already remediated, left un-mirrored on the native list surfaces.
- **Fix (read-only):** Point the selection-highlight brushes at the ADR-026 purple tokens (already defined) and add the existing #8A93A3 lift-ring + per-hue ink pin to the picker/detail kind badges, mirroring the graph layer's KIND_BORDER treatment. Token/brush reuse only, no write path.
- **Subsumes:** rootpicker-2, gap-4, rootpicker-4, workspace-6, connect-4

### Lever 5 - Meaningful counts leak into caption-opacity / translucent fills on the Audit ring and tiles

- **Problem:** The health-ring inner 'donut hole' disc paints the translucent CardBackgroundBrush, so the band hue bleeds through behind the score/band text; the '/ 100' denominator and the Warnings-tile '+ n info' sub-count render at caption opacity 0.6 — muting the always-present numeric fallback the ring's WCAG 1.4.1 story depends on. The same caption-opacity dimming hits Plan's Add-membership hint and Gap's row subject name.
- **Code:** src/App/Views/AuditView.axaml:149-158 (inner disc CardBackgroundBrush + '/ 100' caption), tile '+ n info' sub-count; src/App/Views/PlanView.axaml:141 (caption hint); src/App/Views/GapView.axaml (row subject caption)
- **Impact:** The most-screenshotted attestation artifact (the health ring) can fail 4.5:1 where the band colour shows through the text, and meaningful counts sit near the contrast floor — undercutting the ADR-030 honesty-that-travels intent at the presentation layer.
- **Fix (read-only):** Give the inner disc an opaque page/card fill (or a solid neutral) so no band hue bleeds behind text; promote the '/ 100', '+ n info', and the two dim hints from caption/0.6 to a tier that clears 4.5:1 in both themes (verify with tools/check-contrast.ps1). Read-only styling, no score-math or write change (ADR-030's band gate untouched).
- **Subsumes:** audit-1, audit-4, audit-2, plan-3, gap-5

## Findings by surface

### connect (5)

- **[connect-1]** _(High)_ "Advanced — target a specific domain or DC" trigger does not read as an interactive control (looks identical to the two helper lines above it)
  - **Fails:** ui-checklist "Connect step" — Advanced targeting disclosure (ADR-031, #189) must read as a disclosure the auditor can act on; WCAG 1.4.1 Use of Color / affordance (control identity must not rely on a single channel the user can't perceive)
  - **Evidence:** src/App/Views/ConnectionView.axaml:65-70 (the ToggleAdvanced Button wraps a plain `Classes="secondary"` TextBlock, Background=Transparent, BorderThickness=0, no chevron, no link color, no underline, Padding=0); artifacts/ui/connection-idle-1280x720.png (the "Advanced — target a specific domain or DC" line is visually indistinguishable from the two helper lines directly above it)
  - **Rationale:** The only clickable disclosure on the card is styled byte-for-byte like the static hint copy around it (same `secondary` class = 12px, Opacity 0.7, theme default ink). There is no expand chevron, no link/accent color, no underline, and no rest-state border — the sole cue that it opens fields is that its wording sounds like an action. Compare the codebase's own `Button.hyperlink` style (App.axaml:26-32: Cursor=Hand + link Foreground), which the disclosure does not use. An auditor scanning the card will not discover the ADR-031 targeting capability, defeating the ADR's stated goal of surfacing the already-built server/baseDn reach. A leading ▸/▾ chevron and/or accent/hyperlink treatment (a redundant non-prose channel) is needed.
- **[connect-3]** _(Medium)_ Advanced fields have no persistent visible labels — once filled, the watermark disappears and the value has no caption
  - **Fails:** WCAG 3.3.2 Labels or Instructions (a persistent programmatic/visible label, not a placeholder that vanishes on input); ui-checklist "Advanced targeting disclosure" — "an optional server/DC host field + an optional base-DN field" must remain identifiable
  - **Evidence:** src/App/Views/ConnectionView.axaml:72-75 (both TextBoxes carry only `Watermark="..."`, no `<Label>` / heading TextBlock); artifacts/ui/connect-advanced-dark.png and connect-advanced-light.png show the filled fields as bare "dc1.corp.example" and "OU=Groups,DC=corp,DC=example" with no label distinguishing which is server vs base DN once the placeholder text is gone
  - **Rationale:** Placeholder-as-label is a known accessibility anti-pattern: the moment the auditor types (or a value is restored), the only cue for what each field means disappears. For directory inputs where confusing a DC host with a base DN silently retargets the whole audit, the two fields need a persistent visible label (a `caption`/`secondary` TextBlock above each, or an inline field label), not a disappearing watermark.
- **[connect-2]** _(Low)_ Two stacked helper lines under the Connect button say the same thing (integrated-auth line and TargetLine both name DOMAIN\user), reading as redundant grey clutter
  - **Fails:** ui-checklist "Connect step" — "reads as a deliberate first-run, not a default template (#93)"; the Connect-composition intent of a single quiet card, not stacked repetition
  - **Evidence:** src/App/Views/ConnectionView.axaml:52-54 (CurrentUserContext line: "Uses integrated Windows authentication as {0} — no credentials asked.") and :58-60 (TargetLine, which resolves to "as {CurrentUserContext} against the detected domain" per ConnectionViewModel.cs:80-93); artifacts/ui/connection-idle-1280x720.png shows "Uses integrated Windows authentication as AGDLP\Administrator — no credentials asked." immediately followed by "as AGDLP\Administrator against the detected domain"
  - **Rationale:** In the default (collapsed, blank-fields) state both lines restate the same identity `AGDLP\Administrator`; the TargetLine's "detected domain" adds no new information beyond "Uses integrated Windows authentication" already implies. Three consecutive low-contrast grey lines (auth line, target line, Advanced trigger) sit between the primary button and Demo, making the card feel like stacked fine-print rather than the crafted first-run #93 intends. The pre-bind confirmation (ADR-031 D4) is genuinely useful only once a specific server/baseDn is entered; in the zero-config default it duplicates the line above it and should be suppressed or merged.
- **[connect-4]** _(Low)_ Concentric-ring signature strokes are far below the 3:1 non-text contrast floor (decorative, but the eyebrow at Opacity 0.6 sits near the same ratio)
  - **Fails:** WCAG 1.4.11 non-text contrast (3:1) is the checklist bar; the rings are explicitly decorative (exempt) but the eyebrow "ACTIVE DIRECTORY AUDIT" is meaningful text under WCAG 1.4.3 (4.5:1)
  - **Evidence:** src/App/Views/ConnectionView.axaml:21-28 (ring strokes #14/#16/#0F/#0BFFFFFF over #1b1f27 ≈ 1.03–1.05:1 — intentional per the comment) and :38 (`Classes="eyebrow"` = Opacity 0.6 per Typography.axaml:56-61) rendered over the card composite #14FFFFFF-on-#1b1f27; artifacts/ui/connection-idle-1280x720.png
  - **Rationale:** The rings are a sanctioned decorative aesthetic risk (#93) and are correctly IsHitTestVisible=False, so they are exempt from 1.4.11 — flagging only to note they must stay decorative. The real check is the eyebrow: at Opacity 0.6 the light-grey ALL-CAPS label on the translucent card should be measured against the 4.5:1 text floor (WCAG 1.4.3) by tools/check-contrast.ps1; the light-theme variant (connect-advanced-light.png) shows it noticeably faint. Verify it clears 4.5:1 in both themes rather than assuming the decorative ring language covers it.
- **[connect-5]** _(Low)_ The "runas /netonly" alternate-credentials note is buried inside the collapsed Advanced panel, so the E2 boundary an untrusted-forest auditor needs is invisible by default
  - **Fails:** ui-checklist "Advanced targeting disclosure" — "the OS runas /netonly note replaces an in-app credential prompt"; ADR-031 D2 states this boundary belongs "on the Connect card / README"
  - **Evidence:** src/App/Views/ConnectionView.axaml:77-79 (the runas note is inside the `IsVisible="{Binding IsAdvancedExpanded}"` StackPanel at :71); it renders only when Advanced is expanded — artifacts/ui/connect-advanced-dark.png shows it, connection-idle-1280x720.png does not
  - **Rationale:** ADR-031 D2 places the alternate-account guidance "on the Connect card" as the answer for an auditor who needs different credentials. Because it lives inside the collapsed disclosure (which itself is hard to discover, see connect-1), an auditor who cannot bind with their current token gets no on-card pointer to the sanctioned `runas /netonly` path — they only see a generic connect failure. Given the disclosure is collapsed by default, this note is effectively hidden for the exact persona it serves.

### rootpicker (5)

- **[rootpicker-1]** _(High)_ Load and Back are two identical default-grey buttons — no accent/ghost action hierarchy
  - **Fails:** ui-checklist 'Visual identity > Action hierarchy' (line 118): 'primary actions are accent-filled, secondary are ghost-outlined … no two-identical-grey-buttons ambiguity'
  - **Evidence:** src/App/Views/RootPickerView.axaml:56-57 (both Buttons carry NO Classes); rootpicker-demo-1280x720.png / rootpicker-demo-1920x1080.png (Back and Load render as identical grey pills bottom-corners). Contrast the app-wide vocabulary: ConnectionView.axaml:48 Classes="accent" / :82 Classes="ghost", also used in SettingsWindow/WorkspaceView/DetailPanelView (10 accent/ghost occurrences across 5 views).
  - **Rationale:** Load is the step's primary, selection-gated action (IsDefault="True", command disabled until a candidate is picked) yet is visually indistinguishable from the destructive-ish Back. The checklist's Action-hierarchy item is a cross-cutting bar every adopted view inherits, and this is exactly the 'two identical grey buttons' the Connect step (#93) was reworked to eliminate. RootPicker is the immediate next step in the same flow but was never brought onto the accent/ghost vocabulary. Fix: Classes="accent" on Load, Classes="ghost" on Back — a pure class add, no behavior change.
- **[rootpicker-2]** _(High)_ Selected-row highlight is default-Fluent accent blue, not the ADR-026 brand-purple accent — off-brand and inconsistent with the rest of the app
  - **Fails:** ADR-026 D6 / BrandTokens accent re-base (AccentFillHex #6B5BDF / AccentHex #8B7BFF); ui-checklist 'Mandatory selection gating — selected state' (line 136): 'one row visibly highlighted'
  - **Evidence:** rootpicker-demo-1280x720.png / -tail (the selected AGDLP-Demo and UG_ProjectX rows render in Fluent SystemAccent BLUE ~#0078D4). RootPickerView.axaml:25-28 ListBox sets no selection-brush override, so it inherits FluentTheme's default blue. BrandTokens.cs:245/281 declare the brand purple the app moved to.
  - **Rationale:** The whole app was re-based onto a brand-purple accent (ADR-026 D6, WP2: DEMO pill, primary-button fills, selection/focus rings all purple), but the picker's selected-row band — the single most important visual cue on this mandatory-selection surface — is still stock Fluent blue. The result is a jarring blue selection in an otherwise purple-accented product, and it collides visually with the OU kind badge (#0F6CBD blue). Override the ListBoxItem :selected background with the accent-soft / accent token so the selection cue is on-brand and distinct from the OU badge hue.
- **[rootpicker-3]** _(Medium)_ Picker step has no orientation eyebrow, no candidate-count, and no empty/no-match state — reads as a raw list dump after the crafted Connect card
  - **Fails:** ui-checklist 'Connect composition' first-run-not-a-template standard (line 124, the sibling step's bar) + 'Filter box narrows the list live … clearing restores the full list' (line 138); WCAG-adjacent affordance honesty
  - **Evidence:** RootPickerView.axaml:15 is a bare Classes="title" TextBlock with no eyebrow/subtitle; no TextBlock bound to FilteredCandidates.Count anywhere in the view; grep for 'No candidates|No match|empty' in RootPickerView.axaml returns zero matches. Compare ConnectionView.axaml:37-46 (eyebrow + wordmark + subtitle). rootpicker-demo screenshots show the list flush under the title with no count/scale cue.
  - **Rationale:** The Connect step immediately preceding this was deliberately crafted (eyebrow 'ACTIVE DIRECTORY AUDIT', card, subtitle) so it 'reads as a deliberate first-run, not a default template'. The picker — which must scale to thousands of virtualized candidates (its own XAML comment) — drops all of that: no eyebrow/voice, no 'N candidates' scale cue, and when the live filter matches nothing the ListBox simply goes blank with no 'No candidates match' text, so the user cannot tell an over-narrow filter from an empty directory. Add a compact count/eyebrow line and a no-match placeholder (mirrors the graph controls' #find-no-match honesty).
- **[rootpicker-4]** _(Low)_ Kind-badge fills (DL/UG) fail WCAG 1.4.11 3:1 against the page background — the picker lacks the lift-ring the graph layer required for the same fills
  - **Fails:** ui-checklist cross-cutting bar (lines 7-13): WCAG 2.2 AA 1.4.11 non-text/graphical contrast 3:1; the graph-layer item (line 31) that added a '2px #8A93A3 contrast-lift ring on DL/UG/Computer' for exactly these sub-3:1 fills
  - **Evidence:** Computed white-on-fill passes (DL 6.46:1, UG 6.22:1) so the LABEL is fine, but badge FILL vs page-bg #1b1f27: DL #A14000 = 2.55:1, UG #744DA9 = 2.66:1, Computer #556070 = 2.59:1 (all < 3:1). BrandTokens.cs:46-59 documents these fills as sub-3:1 and 'lifted by the 2px #8A93A3 ring in graph.js' — but RootPickerView.axaml:32-38 draws the badge Border with only Background, no BorderBrush/BorderThickness lift. rootpicker-demo screenshots: the rust DL and purple UG badges' boundary against the dark rows is faint.
  - **Rationale:** The project already ruled (ADR-021 / #90) that these three fills need a #8A93A3 boundary ring to clear the 3:1 graphical-object floor, and applied it on the canvas nodes — but the root-picker badges use the raw fills with no ring, so on an unselected row the badge shape boundary reads below 3:1 on the dark page. The white label is a redundant channel that keeps the badge identifiable (why this is Medium, not High), but the badge-as-graphical-object still misses the same bar the graph nodes were fixed to. Apply the NodeLiftRing (#8A93A3) hairline to DL/UG/Computer badges for parity with the canvas.
- **[rootpicker-5]** _(Low, contested)_ Filter box offers no clear affordance — clearing a long filter requires manual select-all-delete
  - **Fails:** ui-checklist 'Filter box narrows the list live … clearing restores the full list' (line 138) — the restore path exists in the VM but has no UI affordance
  - **Evidence:** RootPickerView.axaml:17-19 is a plain TextBox with only a Watermark; no clear button, no Esc-clears handler (unlike the graph bundle's find input, ui-checklist line 84 'Esc in Find clears+blurs'). RootPickerViewModel.cs:86-103 FilteredCandidates already restores the full list on blank Filter, so the behavior is one gesture away from an affordance.
  - **Rationale:** The live-filter restore is implemented but the only way to invoke it is to manually empty the field. The graph's Find control sets the in-app precedent (Esc clears+blurs); the picker's filter — which the user hits first and most — has neither an inline clear glyph nor an Esc-to-clear handler, an avoidable friction on a surface built for scanning thousands of rows. Low because the core narrow/restore contract works; this is a missing convenience affordance, not a broken one.

### workspace (6)

- **[workspace-1]** _(Medium)_ Sidebar findings rows are the only interactive control whose focus-visible ring can be swallowed by the active-row highlight — ADR-033 covers Buttons but the jump row is a corner-radius-0 templated Button sitting on a selection-highlight band
  - **Fails:** WCAG 2.4.7 Focus Visible / 2.4.11 Focus Appearance; ADR-033 D1 covered set (Button, ListBoxItem)
  - **Evidence:** ViolationsSidebarView.axaml:156-183 — jump Button, Background=Transparent, BorderThickness=0, CornerRadius unset (0), nested in a Grid whose Background is SelectionHighlightConverters.IsActiveToBackground (line 155). ADR-033 D2/D3 places the FocusAdorner Border at Margin=0 edge-hugging on the control's own radius.
  - **Rationale:** ADR-033 is Accepted and settles that native chrome gets a keyboard ring, so this is a fit gap against a settled decision, not a re-litigation. The finding row is the primary keyboard target of the sidebar (Tab through findings, Enter to jump). With no corner radius and no inset from the selection-highlight band, the edge-hugging 2px accent adorner is the hardest ring in the app to perceive — worst on the :active row where the highlight background is closest in tone to the accent. Falsifiable: render a Tab-focused sidebar frame in both themes and apply the ADR-033 pixel guard (ring band reads accent, not the highlight fill) that other controls already get.
- **[workspace-3]** _(Medium)_ Status-bar DockPanel starves the connection-summary line: three right-docked items plus the load progress bar consume width first, and only the summary (the item with ellipsis) has no MinWidth floor
  - **Fails:** docs/ui-checklist.md Workspace status-bar clause 'single dimmed line, no clipping' (line 146) + Global 'Dimmed/secondary text ... still legible at both sizes' (line 195)
  - **Evidence:** WorkspaceView.axaml:263-278 — three items Dock=Right (root:{DN} Margin 24,0,0,0 line 265-266; a 120px IsIndeterminate ProgressBar line 268-269; GraphSummary line 272-275) are laid out first; ConnectionSummary (line 276-277) is the sole fill child and the only one with TextTrimming=CharacterEllipsis. No MinWidth on the summary.
  - **Rationale:** Right-docked-first plus one left fill is the classic DockPanel squeeze: the human-readable connection summary ('connected, n groups loaded — …') collapses to an ellipsis stub while the verbatim DN keeps full width. During a load all three right items AND the 120px bar are present at once — the worst case, at the 1280 width with the 340px rail eating horizontal space. Falsifiable against workspace-demo / workspace-live-graph at 1280 with a long root DN and IsLoading true.
- **[workspace-4]** _(Medium)_ Sidebar 'Why?' button has no resting affordance (transparent, borderless, hover-only) — an interactive control that does not read as a control on a static frame or to a keyboard user
  - **Fails:** WCAG 1.4.1 Use of Color / 1.4.11 non-text UI-component contrast (3:1); the app's own 'reads as an interactive control' bar applied to the rail chevron (checklist line 163)
  - **Evidence:** ViolationsSidebarView.axaml:46-57 — Button.why: Background=Transparent, BorderThickness=0, secondary ink; its ONLY affordance is the :pointerover GhostHoverBrush wash (lines 55-57). At rest it is the bare word 'Why?' with no border/background boundary. Contrast with DetailPanelView.axaml:55-57 CopyDnButton (Classes=ghost, has a resting border).
  - **Rationale:** The 'Why?' control opens the per-finding rationale flyout (#198) and its comment says it must 'stay discoverable but subordinate' — but with zero resting border/background it fails the discoverable half: on the workspace-violations static frame, and to a keyboard-only auditor, it does not present as a control at all, relying entirely on hover. This is the same use-of-color/affordance pitfall the checklist calls out for chips (active state 'NOT color-only', line 364) and demands for the rail chevron ('reads as an interactive control', line 163). A quiet resting outline or a persistent glyph fixes it without competing with the message.
- **[workspace-2]** _(Low)_ Sidebar Export CSV/HTML tooltips omit the read-only 'writes only the file you pick' assurance the twin Audit-screen export buttons are required to carry
  - **Fails:** docs/ui-checklist.md Audit-screen 'Export buttons (WP2)' clause (line 318): tooltips must note the read-only ('writes only the file you pick') behaviour
  - **Evidence:** ViolationsSidebarView.axaml:83-90 — ExportCsvButton ToolTip.Tip='Export findings as CSV', ExportHtmlButton ToolTip.Tip='Export findings as a self-contained HTML file'. The equivalent audit-view export pair (checklist line 318) mandates a read-only tooltip; the sidebar sidebar clause (line 205) only requires 'tooltips present'.
  - **Rationale:** GroupWeaver's core promise is that it never writes AD. The two export surfaces export the SAME report but only the audit-view pair is required to reassure the user the button writes a file, not the directory.
