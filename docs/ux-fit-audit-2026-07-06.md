> Generated by the `fit-audit` workflow (run `wf_2d321d6e-e38`): 9 surfaces, one critic per surface -> two diverse-lens jurors (consumer persona + WCAG/ADR correctness) -> completeness critic -> synthesizer.
> **Read-only / advisory** - every proposed fix is presentation-only; no write path. This doc is the persisted output; levers are candidates for GitHub issues.
> **Caveat:** the `jury:audit:correctness` agent errored mid-run (server error) and did not retry, so the four `audit/*` findings carry only the consumer-persona jury verdict, not the WCAG/ADR-correctness cross-check the other 8 surfaces got. > **Surfaces:** connect, rootpicker, workspace, graph, audit, plan, gap, settings, crosscut &nbsp;|&nbsp; **Findings:** 44 &nbsp;|&nbsp; **Levers:** 5

## Verdict

GroupWeaver's UX fit is fundamentally sound — the declared design system (BrandTokens, Typography.axaml scale, ADR-033 focus ring, ADR-036 action hierarchy) exists and is *mostly* wired up, and several of these findings are literally the audit catching drift from an already-proven fix pattern (the AuditView.axaml:886-894 contrast fix, the Violations-sidebar scroll pattern, the root-picker persistent-label pattern) rather than a missing system. The recurring failure mode is incomplete propagation: each new surface (Connect's ADR-031 Advanced disclosure, Plan's editor column, Gap's changes-list, Settings' Advanced tab, the ADR-032 drift dashboard) ships ahead of the checklist/screenshot-fixture coverage that would have caught its regression, so the same three defect classes (contrast composites, missing focus-visible/keyboard paths, un-classed typography) reappear surface by surface, and a few high-value interactions (Gap's own double-tap CTA, Plan's Rename/Remove row) are outright unreachable at supported configurations. Nothing here contests a settled ADR outright except gap-3, whose "wrong" checklist wording is actually confirmed correct behavior by ADR-024 (Gap's Back returns to Plan, not "explore" — a stale doc string, not a UI bug).

## Top 5 levers (ranked)

### Lever 1 - Severity/contrast signal integrity — the tool's core "what's wrong and how bad" cue fails WCAG and sometimes disappears outright

- **Problem:** The graph's roll-up ring overwrites a node's own severity halo (losing the redundant-coding signal ADR-010 relies on), and five separate surfaces (Audit status pill + info-caveats, graph control cluster + legend, the WebView2-missing banner) fail 1.4.3/1.4.11 contrast in one or both themes — several by reusing a CardBackgroundBrush-on-CardBackgroundBrush composite the same file already diagnosed and fixed once (AuditView.axaml:886-894).
- **Code:** web/graph.js (roll-up ring compositing, control cluster, legend); src/App/Views/AuditView.axaml (status pill, info-caveat blocks); ConnectionView/shell banner (WebView2-missing tint)
- **Impact:** Auditors can lose the primary severity cue on rolled-up nodes, and low-vision/AA users can't read the exact indicators (status, control labels, missing-runtime warning) the tool exists to surface — a WCAG-bar violation on the app's own headline feature.
- **Fix (read-only):** Change the roll-up ring to composite alongside (outer ring) rather than overwrite the node's own halo color; re-apply the already-proven PageBackgroundBrush-based composite fix from AuditView.axaml:886-894 to the pill and info-caveat blocks; resolve the control-cluster panel and legend zero-count text against the same BrandTokens-neutral pairing already used elsewhere; retint the WebView2-missing banner for 3:1 in both themes. All are style/token/JS-render edits — no behavior or persistence changes.
- **Subsumes:** graph-1, graph-3, graph-4, audit-1, audit-2, crosscut-1

### Lever 2 - ADR-033's keyboard-focus-visible rollout is incomplete, including on the app's most-used control

- **Problem:** Root Picker's Filter TextBox — the surface's primary control — ships with no visible focus indicator at all, and the gap is systemic: Audit's chip strip, Gap's flat-command changes-list rows, and Plan's selectable rows are all in the same boat; the one focus ring that does exist elsewhere (Audit's Categories pane) uses the wrong corner-radius token; the minimap has no keyboard path; and the keyboard cheat sheet itself omits the command palette's own key model and the F1 shortcut used to open the sheet.
- **Code:** src/App/Views/RootPickerView.axaml (Filter TextBox); AuditView.axaml (Categories-pane ring, chip strip); GapView.axaml (changes-list rows); PlanView.axaml (Objects/Memberships rows); web/graph.js (minimap); KeyboardHelpWindow.axaml
- **Impact:** Keyboard-only users lose track of focus on the single most-used control in the app (filtering hundreds of candidates) and on several other primary interaction rows — a WCAG 2.4.7/2.1.1 Level-A gap the app otherwise treats as a shipped, tested convention (ADR-033).
- **Fix (read-only):** Attach the same ADR-033 focus-visual-style resource already used elsewhere to the Filter TextBox, chip strip, changes-list rows, and Plan rows; fix the Categories-pane ring's corner radius to match its 6px row; add an arrow-key/Tab pan equivalent for the minimap; add the command-palette key model and F1 entry to KeyboardHelpWindow's existing list. All are resource-attachment and doc-table edits, no new interaction model.
- **Subsumes:** rootpicker-3, missing-2, audit-4, graph-5, crosscut-2, crosscut-3

### Lever 3 - Primary interactions that the UI itself advertises are silently unreachable

- **Problem:** Gap mode's own hover affordance and Unchecked-banner CTA tell the user to double-tap a frontier node to expand it — doing so is a silent dead end. Plan's editor column clips the Rename/Remove action row and hides the entire Findings section with zero scroll affordance at the pinned 1280x720 size, even though the sibling Violations sidebar already ships the fix pattern (ListBottomFadeMask on a scrolling container). Root Picker's picker column isn't centered and drifts to the corner at 1920x1080 (contested by the correctness juror — see rootpicker-1 below).
- **Code:** web/graph.js (Gap double-tap handler for frontier/unexpanded nodes); src/App/Views/PlanView.axaml (editor column layout); src/App/Views/RootPickerView.axaml (picker column alignment)
- **Impact:** Two core workflows are functionally broken at documented/pinned sizes: the Gap Unchecked-banner's entire remediation path never round-trips, and Plan's Rename/Remove/Findings become invisible or unusable exactly when the window is smallest.
- **Fix (read-only):** Route the frontier double-tap to the same nodeExpand round-trip already wired for the hover path; wrap Plan's editor column in the same scroll+fade-mask container pattern already used in the Violations sidebar so nothing clips; verify Root Picker's column centering empirically against the pinned i221 screenshot before touching it (rootpicker-1 is contested). Layout/wiring fixes only.
- **Subsumes:** gap-1, plan-1, plan-2, rootpicker-1

### Lever 4 - Shipped surfaces are outrunning the DoD's own headless-verification gate

- **Problem:** Several already-shipped, ADR-backed surfaces have never been screenshotted or checklist-judged per CLAUDE.md DoD step 2: Settings' Advanced (JSONC) tab, KeyboardHelpWindow, and the entire ADR-032 Audit run-history/drift dashboard. A cluster of specific behaviors (Connect's busy/error-dismissal transitions, Root Picker's back-state reset and virtualization, Settings' File-tab import/export + live re-thread, Plan's add-object/add-membership forms, exported report artifacts) are likewise unverified, so any regression there is currently invisible by construction.
- **Code:** docs/ui-checklist.md (missing sections); tests/GroupWeaver.App.Tests/Screenshots (missing fixtures for Settings-Advanced, KeyboardHelpWindow, Audit drift)
- **Impact:** The project's own Definition of Done can't catch regressions on features it has already shipped and is actively relying on for the current release cadence (ADR-032's drift dashboard in particular) — this is process debt with real regression risk, not a cosmetic gap.
- **Fix (read-only):** Add ui-checklist.md sections + Avalonia.Headless/Playwright screenshot fixtures for the three unaudited surfaces, then add interaction-verification checks ([I]-style) for the nine unverified behaviors, following the DoD step-2 pattern already used for every other adopted view. Test/doc-authoring only — no product code changes required by this lever itself.
- **Subsumes:** settings-1, crosscut-4, missing-1, missing-3, missing-4, missing-5, missing-9, missing-8

### Lever 5 - Newer surfaces skip the already-declared typography/DN-honesty and disclosure conventions

- **Problem:** The Connect step's Advanced disclosure (ADR-031, recently shipped) doesn't read as interactive, and its Server/DC-host and Base-DN fields lose their only label the moment they're filled — both fail conventions the same checklist already models elsewhere (Button.hyperlink; Root Picker's persistent name/DN). Root Picker's own candidate-row Name is unclassed (contested by the correctness juror), Connect's and Gap's directory strings render in proportional type instead of the `dn`/`dn-strong` mono classes Typography.axaml already ships, and the filter watermark undersells that SAM is searchable.
- **Code:** src/App/Views/ConnectionView.axaml (Advanced disclosure toggle, field labels); RootPickerView.axaml (candidate row, filter watermark); GapView.axaml (changes-list tooltip)
- **Impact:** Inconsistent application of an already-built design system makes the newest, most user-visible feature (domain targeting) read as unfinished, and breaks the monospace-honesty convention that lets users trust which text is verbatim directory data.
- **Fix (read-only):** Apply the existing `Button.hyperlink` class to the Advanced toggle; add a persistent (non-placeholder) caption under the two Advanced fields mirroring Root Picker's own pattern; apply the existing `dn`/`dn-strong`/type-scale classes from Typography.axaml to the Connect target line, the candidate-row Name, and Gap's tooltip (swapping the fixed per-kind label for PrimaryDn); extend the filter watermark text to mention SAM. Class/resource wiring only, no new styles.
- **Subsumes:** connect-1, connect-2, connect-3, rootpicker-2, rootpicker-4, gap-2

## Findings by surface

### connect (3)

- **[connect-1]** _(Medium)_ "Advanced" disclosure toggle is visually indistinguishable from static hint text
  - **Fails:** `Button.hyperlink` house pattern (App.axaml:24-36) for exactly this affordance; ui-checklist.md's "reads as an interactive control" bar (line 170, rail chevron clause)
  - **Evidence:** ConnectionView.axaml:65-70 strips every `ghost` visual (Padding=0, Background=Transparent, BorderThickness=0) and wraps a plain `Classes="secondary"` TextBlock — no hyperlink class, no color/cursor change, no chevron. artifacts/ui/connect-advanced-{dark,light}.png show it rendered identically to the static caption lines around it.
  - **Rationale:** A sighted user has no pixel-level cue that the middle line is clickable versus informational; ADR-031 D1 mandates the toggle's existence/behavior but says nothing about its visual affordance, so this is unsettled by any ADR.
- **[connect-2]** _(Low)_ Directory strings in the pre-bind target line use proportional "secondary" type, not the app-wide tabular-mono `dn` face
  - **Fails:** ui-checklist.md line 122 "Monospace honesty" — the same convention Workspace/RootPicker/DetailPanel/Gap/Plan all apply via `Classes="dn"`
  - **Evidence:** ConnectionViewModel.cs:80-95 (`TargetLine`) interpolates the raw host/baseDn into one sentence bound with `Classes="secondary"` only (ConnectionView.axaml:58-60) — no mono class anywhere on the line.
  - **Rationale:** Pre-bind display text, not audit data being triaged/exported — real convention break but low task impact.
- **[connect-3]** _(Medium)_ Advanced Server/DC-host and Base-DN fields carry no persistent label — identity is watermark-only and disappears once filled
  - **Fails:** WCAG 2.2 SC 3.3.2 Labels or Instructions
  - **Evidence:** ConnectionView.axaml:72-75 — both TextBoxes have only a Watermark, no adjoining label, no AutomationProperties.Name; screenshots show both fields filled with nothing left identifying which is which.
  - **Rationale:** A returning auditor who re-expands Advanced sees two filled boxes with no way to tell host from base-DN — textbook placeholder-as-label failure; a real functional risk (wrong DC/base silently narrows the audit scope).

  - **Fails:** ui-checklist.md Global "Centered layouts ... stay centered ... at 1920x1080" (line 203)

- **[rootpicker-2]** _(Medium, contested)_ Candidate row's Name TextBlock carries no typography class

  - **Rationale:** Doesn't contradict ADR-033 (never claimed to cover TextBox) but undercuts its stated "close the native-chrome WCAG 2.4.7 gap" goal on exactly the primary control of this surface.
- **[rootpicker-4]** _(Low)_ Filter watermark undersells what the box actually searches (SAM is silently omitted)

### graph (5)

- **[graph-1]** _(High)_ Roll-up ring silently overwrites a node's own severity halo
  - **Fails:** ui-checklist.md "Roll-up cue" (line 52) + "Severity survives lazy expand" (line 51) + WCAG 1.4.1

- **[graph-4]** _(Medium)_ Legend zero-count text fails WCAG 1.4.3 in both themes
  - **Fails:** ui-checklist.md bar statement (1.4.3, 4.5:1) applied to line 45 (live per-kind count)
  - **Evidence:** index.html:89 `.count.zero` color vs `--gw-chrome-count-zero` (graph.js:106/132) composites to ~2.8:1 dark / ~2.4:1 light.

- **[graph-5]** _(Low)_ Minimap click/drag-to-pan has no keyboard equivalent
  - **Fails:** WCAG 2.1.1 (Keyboard, Level A)
  - **Evidence:** graph.js:399-422 binds only mouse events; index.html:368-370 has no tabindex on #minimap.
  - **Rationale:** Real Level-A gap, but Find/Fit/Zoom keyboard shortcuts provide a partial navigation substitute — hence Low, not Medium/High.

  - **Rationale:** This is ADR-011 D4's deliberate design (separate Text targets, never interpolated, to keep untrusted control-chars from composing into one string, #45) — the checklist prose describes that two-part display loosely; a minor wording nit, not a shipped-vs-documented defect.
- **[settings-4]** _(Low, contested)_ Advanced (JSONC) editor escapes ordinary punctuation the same window shows in plain text elsewhere
  - **Fails:** ui-checklist.md:120 cross-cutting bar + the Advanced tab's own "never executed/formatted/interpolated" design intent (ADR-011 D4)
