## Verdict

The never-inspected surfaces hold up worse than the app proper: the in-app experience remains disciplined, but the exported artifacts — the one surface that leaves the operator's machine and represents GroupWeaver to third parties — carry real, reproducible defects rather than mere evidence gaps. The findings CSV shreds comma-bearing DNs in its unquoted UncheckedDns appendix and mojibakes non-ASCII names on Excel double-click (fatal for the German-AD home market the demo's all-ASCII fixture is structurally unable to catch); the emitted plan .ps1 promises idempotence but guards with Get-ADGroupMember, which the repo's own lab notes document as throwing on FSP-bearing groups, so under its own Stop preference the script aborts mid-run on exactly the real directories it targets; and the checklist's ASCII-only clause for that script is unenforced end-to-end. DPI verdicts split: the 2x web captures are largely healthy (one blurry minimap, one shipped "Unck" non-word in the legend key), but native chrome has zero evidence above 100% while several views hard-code scale-hostile fixed geometry — the three new checklist scaling items are currently unjudgeable, which is itself the finding. The empty/error-state matrix and tab-order review confirm the harness has drifted from its own evidence contract (byte-identical workspace fixtures voiding three [S] tags, a Settings modal that tabs footer-first, a help sheet clipping 8 of 18 rows at its default size). Overall fit: the product's read-only honesty story is intact in-app but is currently betrayed at its edges — at the export boundary and at the evidence boundary — and both are cheap to repair.

## Top 5 levers (ranked)

### Lever 1 - Make the findings CSV/HTML exports survive first contact with Excel and real renderers

- **Problem:** The CSV appends an unquoted, comma-laden UncheckedDns appendix (ragged rows, breaks RFC-4180 parsers and Excel), is UTF-8-without-BOM (Excel double-click decodes as ANSI, mojibaking every non-ASCII name for the German-AD target audience), joins multi-DN cells with an unescapable bare semicolon, and the HTML sets ink colors without paired backgrounds or a color-scheme declaration (WCAG F24; forced-dark UAs invert the AA-verified palette), plus hardcoded plurals and missing th scope attributes.
- **Code:** src/Core/Export/ViolationReportExporter.cs (CSV + HTML emission; GapReportExporter.cs shares the pipeline per ADR-013)
- **Impact:** The export is the product's ambassador — the one artifact an AD admin forwards to a colleague or auditor. Today it silently corrupts data for any non-ASCII directory and any DN containing a comma, and can render illegibly under dark-preference user agents; the all-ASCII demo baseline can never evidence any of this.
- **Fix (read-only):** In ViolationReportExporter: quote/escape the UncheckedDns appendix rows (or fold them into the rectangular table with a Section column), emit a UTF-8 BOM (or document sep=/TSV — either way pin the decision as the checklist demands), escape or restructure the multi-DN sub-delimiter, and in the HTML template declare color-scheme: light plus background-color paired with every color (F24), fix plural forms, add th scope attributes. Pure output-formatting changes; zero directory interaction.
- **Subsumes:** connect/connect-1, connect/connect-2, connect/connect-3, workspace/ws-exp-1, workspace/ws-exp-3, graph/export-csv-1, graph/export-csv-2, graph/export-html-1, audit/audit-1, audit/audit-2, audit/audit-8, audit/audit-9, plan/plan-2, plan/plan-6, plan/plan-7, gap/gap-2, gap/gap-3, gap/gap-4, gap/gap-12, settings/exports-1, settings/exports-2, settings/exports-3, crosscut/crosscut-2, crosscut/crosscut-3, crosscut/crosscut-8, completeness/missed-5

### Lever 2 - Fix the emitted plan script so it can actually run on a real directory

- **Problem:** The generated .ps1's idempotence guards call Get-ADGroupMember — which the repo's own lab-environment.md documents as throwing 'unspecified error' on any group containing an unresolvable FSP (and on >5000-member groups) — under Set-StrictMode + $ErrorActionPreference='Stop', so the script aborts mid-run on precisely the messy real-world directories it exists for. Separately, the checklist's 'ASCII-only, PS 5.1-parses-clean' clause is structurally unenforced: non-ASCII group names flow into the emitted UTF-8-no-BOM file, which PS 5.1 mis-decodes into a parse error.
- **Code:** src/Core/Plan/PlanScriptExporter.cs (guard-statement templates and file encoding/character policy)
- **Impact:** An admin who trusts the header's 'idempotent — existence checks guard every create' promise gets a half-applied plan and a stack trace on any FSP-bearing or non-ASCII-named directory — the worst possible failure mode for a read-only tool's sole outbound instrument.
- **Fix (read-only):** Change the emitted guard text to read the raw 'member' attribute (Get-ADGroup -Properties member) instead of Get-ADGroupMember — exactly the idiom the lab notes prescribe — and enforce the ASCII contract at generation time (escape or transliterate non-ASCII into the PS 5.1-safe form, or emit with BOM and update the checklist clause). This changes only the text the generator writes; GroupWeaver itself still never touches AD.
- **Subsumes:** connect/connect-4, workspace/ws-exp-2, graph/export-ps1-1, audit/audit-10, plan/plan-1, gap/gap-5, settings/exports-4, crosscut/crosscut-7, completeness/missed-2

### Lever 3 - Repair keyboard focus order and focus continuity across native chrome

- **Problem:** Settings tabs footer-first (Apply/Save/Cancel precede all content in markup order); Audit triage buttons collapse under the focused element leaving focus in the void; Plan-mode entry/exit and Connect→RootPicker transitions drop focus with no parking; findings/plan/gap lists are O(2N) tab gauntlets with no TabNavigation containment; the help sheet clips 8 of 18 rows at its default 560x480 with no keyboard scroll path; the WebView graph HWND has no verified Tab-in/Tab-out handoff; the read-only pill's trust message is hover-only; the 18 matrix ComboBoxes are accessibly indistinguishable even after #320.
- **Code:** src/App/Views/SettingsWindow.axaml (markup order / TabIndex), src/App/Views/KeyboardHelpWindow.axaml (default size + Focusable scroll host), AuditView triage button visibility handling, PlanView/GapView/workspace list KeyboardNavigation directives
- **Impact:** Keyboard-only and AT users — the audience the ADR-033/#230 focus work explicitly serves — face a modal that starts at the wrong end, actions that eat their focus, and lists that cost dozens of Tab presses to cross; the compounding effect materially undercuts the already-filed #315/#317-#322 remediation.
- **Fix (read-only):** Reorder SettingsWindow markup (or set TabIndex) so content precedes the footer; on triage-button collapse, move focus to the row or next actionable sibling; park focus on step transitions per the ADR-022/#230 house pattern; add KeyboardNavigation.TabNavigation="Once"/Contained on repeating lists; enlarge or make scrollable-by-keyboard the help sheet (Focusable ScrollViewer + visible cue); give matrix ComboBoxes row/column-composed AutomationProperties.Name. All view-layer/axaml changes, no data writes.
- **Subsumes:** settings/settings-1, settings/settings-2, audit/audit-3, audit/audit-7, plan/plan-3, plan/plan-9, workspace/ws-kbd-1, workspace/ws-kbd-2, gap/gap-10, graph/tab-1, rootpicker/rootpicker-4, connect/connect-7, connect/connect-8, crosscut/crosscut-1, crosscut/crosscut-5, crosscut/crosscut-10, crosscut/crosscut-11

### connect (9)

- **[connect-1]** _(Medium)_ Findings CSV UncheckedDns appendix emits unquoted comma-bearing DNs — ragged rows that Excel splits across cells
  - **Fails:** ui-checklist.md:513-517 'Findings CSV: opens in Excel without mangling — delimiter/quoting correct for DNs containing commas (escaped DNs stay verbatim)'
  - **Evidence:** artifacts/ui/wp3/exports/findings.csv:22-24; src/Core/Export/ViolationReportExporter.cs:285-296 and :331 (comma deliberately excluded from LineBreakingQuoteTriggers)
  - **Rationale:** The appendix line 'CN=Domain Admins,CN=Users,DC=weavedemo,DC=example' is unquoted, so a plain Excel open splits one DN into four cells (the DN does NOT stay verbatim) and the file becomes a non-rectangular CSV that strict RFC-4180 parsers (pandas, Power Query typed import) reject or mis-shape. The exclusion is code-deliberate and test-pinned but decided only in a code comment, not an Accepted ADR — the checklist's own bar ('escaped DNs stay verbatim') fails on exactly the DN-with-comma case it names. Quoting the appendix cells like the row cells (EncodeCell) would fix it without touching the findings table.
- **[connect-2]** _(Medium)_ Findings CSV is UTF-8 without BOM — Excel double-click decodes it as ANSI, mangling any non-ASCII name
  - **Fails:** ui-checklist.md:513-517 'opens in Excel without mangling — encoding deliberate (UTF-8; BOM decision explicit)'
  - **Evidence:** hexdump of artifacts/ui/wp3/exports/findings.csv (first bytes 52 75 6c — no EF BB BF); src/App/Export/AuditExportService.cs:98-101 (UTF8Encoding(false) codebase discipline)
  - **Rationale:** The no-BOM discipline is right for .ps1/.json but CSV is the one format where desktop Excel needs the BOM to pick UTF-8 on a double-click open; without it Excel falls back to the ANSI codepage and umlauts in group/user names mojibake. The demo artifact is all-ASCII so this frame can't show the damage, but German-localized directories are this product's documented home turf (.claude/rules/lab-environment.md AD quirks). The checklist demands the BOM decision be explicit FOR the Excel-open scenario; the current decision is explicit only as a generic file-writer discipline. Emitting the BOM solely in the CSV writer (or documenting a deliberate no-BOM + import-wizard stance in an ADR) closes it.
- **[connect-4]** _(Medium)_ plan-export.ps1 idempotency guards use Get-ADGroupMember, which throws on groups containing unresolvable FSPs — the emitted script aborts mid-run on real directories
  - **Fails:** ui-checklist.md:525-528 'Plan export .ps1 … object/membership statements mirror the draft plan 1:1, parses clean' + the artifact's own header promise 'idempotent - existence checks guard every create' (plan-export.ps1:4-5)
  - **Evidence:** artifacts/ui/wp3/exports/plan-export.ps1:11 ($ErrorActionPreference='Stop') with :32 and :35 (Get-ADGroupMember -Identity … | Where-Object); .claude/rules/lab-environment.md AD-quirks bullet ('Get-ADGroupMember throws "unspecified error" on any group containing an unresolvable FSP — always read the raw member attribute instead')
  - **Rationale:** The repo's own environment rules document the exact failure: on a target group that already exists and contains an orphaned FSP (common in aged real directories), Get-ADGroupMember throws; with ErrorActionPreference Stop the script dies mid-run, leaving a partially-applied plan — the opposite of the idempotence the header advertises. Reading the group's raw 'member' attribute (Get-ADGroup -Properties member) for the existence check keeps the script inert-text/read-only-product semantics (GroupWeaver still never runs it) while making the emitted artifact robust. ASCII-only/CRLF/no-BOM/PS5.1-parse all verified clean — this is the sole defect in the script.
- **[connect-6]** _(Medium)_ Connect card has no vertical reflow path — fixed 420px centered card with no ScrollViewer, and zero scaled-native evidence exists to disprove clipping at 150-200%
  - **Fails:** ui-checklist.md:221-227 (Native chrome at 150%/200% display scale; long-string resilience at scale) / WCAG 1.4.4 Resize Text, 1.4.10 Reflow
  - **Evidence:** src/App/Views/ConnectionView.axaml:11 (bare Panel root), :30-34 (Border Width=420, VerticalAlignment=Center, no ScrollViewer anywhere in the view), :73-84 (Advanced panel adds ~5 more rows), :97-100 (error block adds more); no connect capture at any scale >1.0 exists (artifacts/ui inventory + README-evidence.md §4)
  - **Rationale:** At 200% display scale a 1366x768 desktop yields a 683x384 logical viewport; the card's stacked content (eyebrow + display wordmark + subtitle + 2 buttons + 3 secondary paragraphs, taller still with Advanced expanded or the error visible) exceeds that height, and with VerticalAlignment=Center and no ScrollViewer both the eyebrow/wordmark top and the error block bottom clip with no recovery gesture. The checklist itself concedes headless renders 1.0 only and prescribes the windowed capture-window.ps1 path on a scaled desktop — that evidence has never been produced for any connect state, so the risk is currently unfalsified in either direction. Wrapping the Panel in a ScrollViewer is a no-visual-change fix at 100%.
